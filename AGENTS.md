# Repository Guidelines

## 项目结构与模块组织
- 根目录下的 `main.go` 是 QQBot 服务入口，负责 WebSocket 服务、消息处理、配置加载与模型路由。
- `internal/codexcli/` 用于对接 Codex CLI，`internal/openaiutil/` 用于处理 OpenAI 兼容接口相关逻辑。
- 配置文件位于 `configs/`，其中 `configs/config.example.json` 作为示例，`configs/config.json` 用于本地实际运行。
- 运行后会生成 `qqbot.log` 与 `user_memory.json`；其中 `user_memory.json` 同时维护用户历史与群级配置，清理前请先备份。
- 模型提示词可通过根目录的 `gpt.txt`、`claude.txt`、`grok.txt` 配置，缺省时回退到内置默认值。

## 构建、测试与开发命令
- `go run .`：以默认配置启动 QQBot 服务。
- `go build ./...`：检查项目是否可以完整编译。
- `go test ./...`：运行 Go 测试；如当前尚未补齐测试，可先用于验证测试入口是否正常。
- `go fmt ./...`：格式化 Go 代码，保持统一风格。
- 运行期命令：`/clear` 清空私聊记忆；`/bv on|off` 与 `/setall <模型>` 管理群聊解析与默认模型；群聊直接 `@机器人` 输入文本将触发默认对话。

## 编码风格与命名约定
- 遵循 Go 官方风格，提交前执行 `go fmt ./...`。
- 未导出标识符使用 `camelCase`，导出标识符使用 `PascalCase`，常量命名保持简洁清晰。
- 配置项与路径常量集中定义，敏感密钥优先通过环境变量或本地配置文件注入，不要硬编码到代码库中。
- 复杂逻辑可添加简洁中文注释，优先解释约束、边界条件与设计意图。

## 测试规范
- 测试文件使用 `*_test.go` 命名，并与被测代码放在同目录。
- 优先覆盖关键路径：模型选择、消息分段、命令处理、外部接口异常与重试逻辑。
- 对外部服务调用应尽量做隔离，避免测试直接依赖真实第三方接口。

## 提交与合并请求准则
- 仓库尚未记录 Git 历史，创建提交时请采用 `类型: 模块 - 摘要`（例如 `feat: bot - 支持消息分段发送`）并在正文列出要点与验证方式。
- Pull Request 描述需包含：变更背景、实现概览、测试结果（含命令输出摘要）以及必要的日志片段；若涉及敏感配置变更需额外列出回滚方案。

## 安全与配置提示
- 所有 API Key、JWT、代理地址必须从环境变量或本地配置注入，切勿提交到版本控制。
- 部署到公网前请确认第三方模型端点、代理与 WebSocket 服务均可达，且日志中不包含用户隐私信息。
